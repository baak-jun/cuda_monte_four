## 📌 1. 현재 코드베이스 상태 및 분석 요약

*   **비트보드 (board.hpp)**: 64비트 정수 두 개(`black`, `white`)에 6행 7열 보드를 7비트 스트라이드(6행+1 sentinel)로 묶어 비트 연산(`has_four`)으로 초고속 승리 판정.
*   **완전 탐색 (exhaustive.cpp)**: 기본 DFS 백트래킹을 이용해 모든 상태 순차 탐색.
*   **CPU Negamax Solver (perfect_solver_cpu.cpp)**: 알파-베타 가지치기와 트랜스포지션 테이블(메모이제이션 해시 캐시)을 사용하여 중복 탐색 제거 (매우 빠름).
*   **GPU Monte Carlo (monte_carlo_cuda.cu)**: 1,000만 번 시뮬레이션을 단 **0.12초** 만에 처리 (초당 8,000만 롤아웃의 성능, GPU가 압도적으로 효율적).
*   **CPU-GPU Hybrid Solver (perfect_frontier_cuda.cu)**: CPU가 프런티어 노드를 늘리고 GPU 스레드가 캐시 없이 각 리프의 서브트리를 전체 탐색. 
    *   **문제점 발견**: GPU 내부의 캐시(트랜스포지션 테이블) 부재로 인해 깊이 15 이상에서 CPU보다 현저히 느려지며 타임아웃 발생.

## 💡 2. Gomoku (오목) 확장 제안

현재 커넥트 포는 선택지(분기 계수)가 최대 7개로 작아 CPU 캐시로도 완벽히 커버되지만, 오목으로 확장할 때 GPU의 성능적 효율성이 극대화됩니다.

1.  **분기 계수 대폭발 ($7 \rightarrow 225$)**:
    *   오목(15x15 판 기준)은 첫 수에 225칸의 선택지가 존재하여 깊이가 조금만 깊어져도 CPU는 기하급수적 계산량 폭발로 다운됩니다.
2.  **GPU의 동시 평가 우위**:
    *   225개의 모든 후보 위치에 대한 병렬 롤아웃 시뮬레이션(Monte Carlo)을 GPU 코어가 동시에 일괄 실행(Batch Evaluation)하므로, 데이터가 커질수록 GPU의 시간 가속비(Speedup Factor)가 극적으로 증가합니다.
3.  **시스템 연구 주제 확장**:
    *   225칸을 표현하기 위해 64비트 한계를 넘어 `uint64_t board[4]`(256비트)로의 비트보드 데이터 확장과 이를 처리하는 GPU 멀티워드 비트 판정 알고리즘 설계.

## ⚡ 3. Shared Memory (공유 메모리) 맛있게 쓰기 전략

단순 롤아웃은 개별 스레드 연산이라 캐시가 필요 없지만, 다음 구조를 도입하면 공유 메모리를 아주 스마트하게 활용할 수 있습니다.

### ① 블록 레벨 초고속 캐시 (Block-Level Transposition Cache)
*   동일 블록 내 128~256개 스레드는 비슷한 서브트리 경로를 탐색하므로 겹치는 노드가 많습니다.
*   공유 메모리에 작은 해시 맵(4KB~8KB)을 잡아두고 블록 내 스레드가 L1 캐시처럼 읽고 써서 중복 연산 노드를 빠르게 가지치기(Cutoff)합니다.

### ② 오목판 협동 평가 (Cooperative Evaluation)
*   15x15 오목 판 자체를 블록 공유 메모리에 1회만 로드합니다.
*   블록 내 스레드들이 역할을 쪼개어 가로/세로/대각선 방향의 오목 분석 점수를 개별 계산한 뒤, **Parallel Reduction**으로 합산하여 최종 국면 가치를 판단합니다.
